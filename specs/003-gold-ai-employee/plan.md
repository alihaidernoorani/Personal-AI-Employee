## Summary

Build a locally-hosted autonomous AI Employee that monitors Gmail, WhatsApp, the filesystem, and bank transactions via Python watcher daemons; reasons over events using Claude Code and agent skills; executes approved actions through domain-specific MCP servers (Email, Browser, Social, Odoo); enforces human approval for all sensitive actions via a file-movement gate; generates a weekly CEO Briefing; and maintains a full audit log. The Ralph Wiggum Stop hook provides multi-step task persistence. All state is carried in Obsidian vault markdown files — no external database.

## 1. High-Level Architecture

The system follows a strict **Perception → Reasoning → Action → Logging** pipeline:

| Stage | Component | Input | Output |
|-------|-----------|-------|--------|
| Perception | Watcher daemons | External sources (Gmail, WhatsApp, filesystem, bank) | `Needs_Action/*.md` |
| Reasoning | Claude Code + skills | `Needs_Action/*.md` + `Company_Handbook.md` | `Plans/PLAN_*.md` + `APPROVAL_*.md` or direct MCP call |
| Gate | ApprovalWatcher + human | File in `Pending_Approval/` | File moved to `Approved/` or `Rejected/` |
| Action | MCP servers (via execute-plan skill) | Approved plan | Real-world action (email sent, post published, Odoo record created) |
| Logging | All MCP calls | Action result | `Logs/YYYY-MM-DD.json` NDJSON entry |
| State close | Skill | Completed action | Task moved to `Done/`; `Dashboard.md` updated |

The Obsidian vault is the **single communication bus** between all components. No component holds shared in-memory state — every state transition is a file move.

## 2. Component Design

### 2.1 Watchers Layer (Python daemons)

**Pattern**: All watchers extend `BaseWatcher` (`watchers/base_watcher.py`). The base class provides the polling loop, error handling, idempotency check, and `PollingObserver` for NTFS mounts.

**BaseWatcher contract**:
- `check_for_updates() → list[item]` — polls source, filters already-processed IDs
- `create_action_file(item) → Path` — writes structured `.md` to `Needs_Action/`
- `run()` — infinite poll loop with configurable `check_interval`
- On any exception: writes `Needs_Action/ERROR_<watcher>_<timestamp>.md` and continues

**Gmail Watcher**
- Polls Gmail API for unread, important messages at 120-second intervals
- Extracts: From, Subject, Date, snippet, message ID
- Writes `Needs_Action/EMAIL_<message_id>.md`
- Tracks processed message IDs in `scripts/processed_gmail.json`
- Graceful degradation: if Gmail API is down, outgoing-email requests are buffered in `scripts/email_outbox_queue.json` for delivery when restored

**WhatsApp Watcher**
- Launches exactly one Playwright `launch_persistent_context` browser at startup; session cookie stored at `WHATSAPP_SESSION_PATH` from `.env`
- Polls at 30-second intervals; scans unread chats for configured keywords
- Writes `Needs_Action/WHATSAPP_<chat_id>_<timestamp>.md`
- Session guard: checks for existing browser PID before launching; if alive, reuses it
- On session expiry: attempts single re-authentication; on failure, writes `ERROR_*.md` and pauses

**Filesystem Watcher**
- Uses `PollingObserver(timeout=3)` on `AI_Employee_Vault/Inbox/`
- On new file: copies file, writes `Needs_Action/FILE_<filename>_<timestamp>.md` with metadata
- No idempotency file needed (filesystem events are intrinsically one-shot per file arrival)

**Finance Watcher**
- Polls configured bank source (CSV export path or open-banking API endpoint) at 300-second intervals
- Appends new transactions to `Bank_Transactions.md` (append-only, never rewrites existing rows)
- Writes `Needs_Action/FINANCE_<transaction_id>.md` for each new transaction
- Tracks processed transaction IDs in `scripts/processed_finance.json`

**Approval Watcher**
- Polls `AI_Employee_Vault/Approved/` every 30 seconds
- On new `APPROVAL_*.md` file: reads plan reference, invokes `execute-plan` skill
- Polls `AI_Employee_Vault/Rejected/` — no action taken, files remain as permanent archive

### 2.2 Orchestrator (Central Controller)

`orchestrator.py` is a single Python process responsible for:

1. **Process management**: Starts all five watcher processes as subprocesses; records PIDs
2. **Watchdog**: Checks watcher liveness every 60 seconds; auto-restarts up to 3 times per hour; after 3 restarts, writes `ERROR_WATCHDOG_<process>_<timestamp>.md` and sends human alert
3. **Cron scheduler**: Manages all scheduled tasks (Sunday 23:00 CEO briefing; daily 08:00 catch-up sweep; 90-day log cleanup) using the `schedule` library
4. **Vault health check**: Checks `VAULT_PATH` mount availability on startup and every 5 minutes; if unavailable, pauses all watchers, writes `ERROR_VAULT_UNAVAILABLE.md` to temp folder, retries every 30 seconds
5. **Graceful shutdown**: On SIGTERM/SIGINT, cleanly terminates all child processes

### 2.3 Claude Code (Reasoning Engine)

Claude Code is not a daemon — it is invoked by the orchestrator (via cron or file event) and runs skills. All reasoning is encapsulated in skill files under `.claude/skills/`.

**Existing skills (do not rebuild)**: `process-needs-action`, `execute-plan`, `linkedin-post`, `reasoning-loop`, `browsing-with-playwright`, `skill-creator`.

**process-needs-action skill** (core loop):
1. Read `Company_Handbook.md` and `Business_Goals.md`
2. Scan `Needs_Action/` for unclaimed files
3. Claim first unclaimed file (move to `In_Progress/local_agent/`)
4. Generate `Plans/PLAN_<timestamp>.md` with action list and approval flag
5. Evaluate permission boundary (FR-017/FR-018):
   - Auto-approve → call `execute-plan` skill directly
   - Requires approval → write `Pending_Approval/APPROVAL_<timestamp>.md`
6. Update `Dashboard.md`
7. Write audit log entry
8. Move source task to `Done/`
9. Repeat until `Needs_Action/` is empty

**ceo-briefing skill**:
1. Read `Business_Goals.md` (targets, KPIs, subscription audit rules, Active Projects with deadlines)
2. Read `Bank_Transactions.md` (all transactions for period)
3. Read `Done/` folder (completed tasks for period — used for Completed This Week section)
4. Scan `Needs_Action/` + `In_Progress/` for stalled tasks (>24 h without state change — used for Bottlenecks section)
5. Read `Logs/YYYY-MM-DD.json` for the period (social media post entries — used for Social Media Summary section)
6. Generate `Briefings/YYYY-MM-DD_Monday_Briefing.md` using the defined CEO Briefing Schema, including all seven mandatory sections: Revenue Summary, Completed This Week, Bottlenecks, Cost Optimisation Suggestions, Upcoming Deadlines (from Business_Goals.md Active Projects), Social Media Summary
7. Write audit log entry

**execute-plan skill**:
1. Read the referenced `Plans/PLAN_*.md`
2. For each action checkbox: call the appropriate MCP tool
3. On success: check off the action, write audit log entry
4. For `social_post` action type specifically: after calling `post_<platform>`, call `get_post_summary` and log engagement data as a second audit entry
5. On failure: apply retry logic (3 attempts, exponential backoff); on exhaustion, write `ERROR_*.md`
6. After all actions: move task to `Done/`, update `Dashboard.md`

**linkedin-post skill**:
1. Read `Business_Goals.md` + `Company_Handbook.md`
2. Generate a 100–300 word lead-generation post
3. Write `Plans/PLAN_LINKEDIN_<timestamp>.md`
4. Write `Pending_Approval/APPROVAL_LINKEDIN_<timestamp>.md` (always requires approval)

### 2.4 MCP Servers (Action Layer)

Each MCP server communicates with Claude Code via **stdio transport**. Servers are registered in `.mcp.json` at the project root (not `~/.config/claude-code/mcp.json`). Each runs as an independent process — starting/stopping one does not affect others.

**Email MCP already exists** (`mcp-servers/email-mcp/server.py`) — Python, SMTP/IMAP-based. Do not rebuild. Extend with `queue_email` and `flush_queue` tools for graceful degradation if not already present.

See **Section 5 (MCP Design)** for full interface contracts.

### 2.5 Odoo Integration Layer

The `odoo_mcp/server.py` Python MCP server wraps all Odoo 19+ JSON-RPC API calls. See **Section 7** for full integration design.

### 2.6 Social Automation Layer

`mcp-servers/social-mcp/server.py` (Python) handles all four platforms. Post and summary generation are separate operations. See **Section 5** for interface design.

### 2.7 Logging & Monitoring

Every MCP action call — regardless of outcome — produces one NDJSON line appended to `Logs/YYYY-MM-DD.json`. The log is the authoritative audit trail.

`Dashboard.md` provides real-time human-readable state counts. It is written only by the `process-needs-action` skill (single-writer rule, FR-040).

Log format exactly matches the audit schema in the spec. Log files older than 90 days are deleted by the orchestrator's scheduled cleanup task.

### 2.8 Watchdog System

Implemented inside `orchestrator.py`. The watchdog runs a liveness check loop:
- Checks each watcher process PID every 60 seconds
- On dead process: restarts, increments hourly restart counter
- Counter resets at the top of each hour
- At 3 restarts/hour: stops auto-restart, writes `ERROR_WATCHDOG_<process>.md`, alerts human
- Vault health check runs separately every 5 minutes

## 3. Data Flow — Task Lifecycle

```
Step 1: DETECTION
  External event (email arrives, WhatsApp message, file drop, bank transaction)
  → Watcher polls source
  → Watcher checks processed_*.json (skip if already seen)
  → Watcher writes Needs_Action/TYPE_<id>_<timestamp>.md
  → Watcher updates processed_*.json

Step 2: CLAIM
  Claude Code (process-needs-action skill) scans Needs_Action/
  → Moves first unclaimed file to In_Progress/local_agent/
  → (Claim-by-move: if another agent already moved it, skip)

Step 3: PLAN
  Skill reads Company_Handbook.md
  → Evaluates task type and permission boundary
  → Writes Plans/PLAN_<timestamp>.md with:
      - Objective
      - Decision rationale
      - Ordered action list ([ ] checkboxes)
      - requires_approval: true/false
      - approval_category

Step 4a: AUTO-APPROVED PATH
  execute-plan skill called directly
  → Calls MCP tool(s)
  → Writes audit log entry
  → Checks off action in PLAN_*.md
  → Moves task to Done/
  → Updates Dashboard.md

Step 4b: APPROVAL-REQUIRED PATH
  Skill writes Pending_Approval/APPROVAL_<timestamp>.md
  → Dashboard.md updated (Pending_Approval count +1)
  → Task remains in In_Progress/local_agent/ (blocked)
  → Human reviews in Obsidian

Step 5: HUMAN DECISION
  Human moves APPROVAL_*.md to:
    Approved/  → ApprovalWatcher detects (within 30 s)
                → execute-plan skill called
                → MCP action executed
                → Audit log written
                → Task moved to Done/
                → Dashboard.md updated
    Rejected/  → File archived permanently
                → Task moved to Done/ with rejection note
                → Audit log written

Step 6: CLOSE
  Dashboard.md updated with final counts
  Audit log entry written for state transition
```

## 4. Folder Architecture

| Folder | Writer | Reader | Rule |
|--------|--------|--------|------|
| `Inbox/` | Human / external drops | Filesystem watcher | Drop zone only; files copied to `Needs_Action/`, originals remain |
| `Needs_Action/` | All watchers | process-needs-action skill | Any file here is claimable; claim = move to `In_Progress/` |
| `In_Progress/local_agent/` | process-needs-action skill | Skills | First mover owns; others skip |
| `Plans/` | Skills | execute-plan skill, human | Append-only; never deleted; linked from approval files |
| `Pending_Approval/` | Skills | ApprovalWatcher, human | Human moves out; skill never moves its own file |
| `Approved/` | Human | ApprovalWatcher | Trigger folder; emptied after processing |
| `Rejected/` | Human | Read-only after write | Permanent archive; never executed |
| `Done/` | Skills, execute-plan | CEO briefing skill | Permanent archive; never deleted |
| `Logs/` | All MCP calls | Human, CEO briefing skill | NDJSON append-only; files deleted after 90 days |
| `Briefings/` | ceo-briefing skill | Human | One file per week; never overwritten |
| `Accounting/` | Finance watcher | CEO briefing skill, Odoo MCP | `Current_Month.md` = alias for `Bank_Transactions.md` content |
| `Bank_Transactions.md` | Finance watcher | CEO briefing skill | Append-only; never rewritten |
| `Dashboard.md` | process-needs-action skill **only** | Human, all skills (read) | Single-writer rule; updated after every state transition |

**Temp folder** (graceful degradation): Configured as `VAULT_TEMP_PATH` in `.env`. Used by watchers when vault path is inaccessible. Files are synced into vault on restoration.

## 5. MCP Design

### 5.1 Email MCP (`mcp-servers/email-mcp/server.py`) — Python

**Domain**: All Gmail interactions
**Transport**: stdio
**Responsibilities**: Send email, create draft, search inbox, queue email locally when service is down

| Tool | Inputs | Output |
|------|--------|--------|
| `send_email` | `to`, `subject`, `body`, `reply_to_id?` | `{ message_id, status }` |
| `create_draft` | `to`, `subject`, `body` | `{ draft_id, status }` |
| `search_inbox` | `query`, `max_results` | `[{ id, from, subject, snippet, date }]` |
| `queue_email` | `to`, `subject`, `body` | `{ queued: true, queue_position }` — used when Gmail API is down |
| `flush_queue` | — | `{ sent: N, failed: M }` — drains `email_outbox_queue.json` |

**Rate limit**: Enforced inside MCP server — max 10 `send_email` calls per hour; returns `429` on breach.
**Error contract**: All tools return `{ error: string, retryable: boolean }` on failure.

### 5.2 Social MCP (`mcp-servers/social-mcp/server.py`) — Python

**Domain**: LinkedIn, Facebook, Instagram, Twitter/X — post publication and activity summary
**Transport**: stdio
**Responsibilities**: Publish approved posts; retrieve post summaries

| Tool | Inputs | Output |
|------|--------|--------|

### 5.5 Communication Pattern

All MCP servers follow the same communication contract:
1. Claude Code calls a tool via the MCP `tools/call` method over stdio
2. MCP server validates inputs, executes action, writes audit log entry to `Logs/`
3. Returns structured JSON result or error object
4. Claude Code interprets result and updates plan checkboxes and vault state
5. On `retryable: true` error: Claude Code retries up to 3 times with exponential backoff (1 s base, 60 s max)

## 6. Ralph Wiggum Loop Design

> **Status: ALREADY IMPLEMENTED.** `scripts/ralph_loop.py` (loop driver), `scripts/check_loop_complete.py` (stop hook), and `.claude/skills/reasoning-loop/SKILL.md` all exist. Gold tier work is to **verify integration** with the finance watcher and CEO briefing skill, not to rebuild the loop.

### 6.1 Stop Hook Architecture

The Ralph Wiggum pattern is implemented as a Claude Code **Stop hook** registered in `.claude/settings.json`:

```
Event: Stop (Claude Code about to exit)
Handler: scripts/check_loop_complete.py
Completion signal: <promise>TASK_COMPLETE</promise> in Claude stdout
State file: In_Progress/claude_code/LOOP_<timestamp>_state.md
```

### 6.2 Iteration Flow

```
Orchestrator
  → Creates state file: AI_Employee_Vault/In_Progress/local_agent/TASK_<id>.state.json
      { "prompt": "<task prompt>", "iteration": 0, "max_iterations": 10,
        "task_file": "<path>", "prior_output": null }
  → Invokes Claude Code with task prompt

Claude Code (iteration N)
  → Works on task
  → Prior output from state file is prepended to context on iterations > 0
  → Completes or partially completes
  → Attempts to exit

Stop Hook checks:
  1. Is task file present in Done/ ?
     YES → write completion to state file → allow exit
  2. Did Claude output <promise>TASK_COMPLETE</promise> ?
     YES → move task to Done/ → allow exit
  3. Has iteration count reached max_iterations?
     YES → write Needs_Action/ERROR_LOOP_MAX_<id>.md → allow exit
  4. Otherwise:
     → Save Claude's current output as prior_output in state file
     → Increment iteration counter
     → Re-inject prompt (blocks exit, Claude continues)
```

### 6.3 Prior Output Context

On each re-injection, the stop hook prepends to the prompt:

```
[LOOP ITERATION N of MAX]
Previous attempt output:
---
{prior_output}
---
Task (continue from where you left off):
{original_prompt}
```

This satisfies FR-028: the agent can reason about why the prior attempt did not complete.

### 6.4 Failure Recovery

| Failure Mode | Recovery |
|---|---|
| Max iterations reached without completion | `ERROR_LOOP_MAX_*.md` written; human review required |
| Stop hook script crash | Claude exits normally (fail-safe); `ERROR_HOOK_CRASH_*.md` written by orchestrator watchdog |
| Task file deleted during loop | Stop hook detects missing file; writes `ERROR_LOOP_MISSING_TASK_*.md`; allows exit |
| Vault inaccessible during loop | Stop hook cannot write state file; allows exit; orchestrator handles vault-unavailable recovery |

## 7. Odoo Integration Plan

### 7.1 JSON-RPC Interaction Model

Odoo 19+ exposes a single external JSON-RPC endpoint. All calls use the same URL and HTTP POST method; the operation is determined by the `method` field in the JSON payload.

**Authentication**: Session-based (via `web/session/authenticate`) using credentials from `.env`. Session cookie is stored in memory for the lifetime of the MCP server process; on expiry, re-authenticates automatically.

**Connection pattern**: The Odoo MCP server maintains a single connection session per process lifetime. No connection pooling required at single-user scale.

### 7.2 Invoice Flow

```
Trigger: Watcher detects billable event → Needs_Action/FINANCE_*.md (type: invoice_needed)
  ↓
process-needs-action skill
  → Reads Finance task file
  → Determines customer (calls odoo_mcp.get_customer)
  → Generates Plans/PLAN_INVOICE_<id>.md
  → requires_approval: true (invoice posting is irreversible)
  → Writes Pending_Approval/APPROVAL_INVOICE_<id>.md

Human approves → Approved/
  ↓
ApprovalWatcher → execute-plan skill
  → Calls odoo_mcp.create_invoice (creates draft)
  → Calls odoo_mcp.post_invoice (marks as posted — irreversible)
  → Writes audit log entry
  → Moves task to Done/
```

### 7.3 Transaction Sync Flow

```
Trigger: Finance watcher appends to Bank_Transactions.md
         Writes Needs_Action/FINANCE_<txn_id>.md (type: transaction_sync)
  ↓
process-needs-action skill
  → Reads transaction data
  → Generates plan: sync to Odoo
  → Auto-approved (transaction sync is non-irreversible categorisation)
  → execute-plan skill called directly

execute-plan skill
  → Calls odoo_mcp.create_transaction with categorised data
  → Calls odoo_mcp.sync_transaction (links bank record to Odoo record)
  → Writes audit log entry
  → Moves task to Done/

CEO Briefing
  → Calls odoo_mcp.list_invoices for period
  → Cross-references with Bank_Transactions.md
  → Reports any unsynced transactions as bottleneck items
```

### 7.4 Odoo MCP Error Strategy

| Error Type | Response |
|---|---|
| Network timeout | Retry 3× with exponential backoff; if all fail → `ERROR_ODOO_<id>.md` |
| Authentication expired | Re-authenticate once automatically; if fails → pause Odoo MCP, write `ERROR_ODOO_AUTH.md` |
| Odoo record conflict | Write `ERROR_ODOO_CONFLICT_<id>.md` with full context; do not retry automatically |
| Odoo unreachable (3+ cycles) | Pause finance sync; CEO Briefing notes sync gap; `ERROR_ODOO_UNAVAILABLE.md` |

## 8. Error Handling Architecture

### 8.1 Retry Strategy

All transient errors across all components use the same exponential backoff:
- Base delay: 1 second
- Multiplier: 2× per attempt
- Max delay: 60 seconds
- Max attempts: 3
- After exhaustion: write `ERROR_*.md` to `Needs_Action/`, do not re-attempt

### 8.2 Error Taxonomy and Responses

| Category | Detection | Immediate Action | Recovery |
|---|---|---|---|
| **Transient** (network, rate limit) | Exception or HTTP 429/5xx | Exponential backoff retry | Auto — up to 3 attempts |
| **Authentication** (expired token, revoked) | HTTP 401/403 | Pause affected watcher/MCP; write `ERROR_AUTH_*.md` | Human resolves; manual restart |
| **Logic** (misparse, wrong action) | Validation failure in skill | Write `ERROR_LOGIC_*.md` with full context | Human reviews and corrects |
| **Data** (corrupted file, missing field) | Schema validation in watcher/skill | Quarantine file to `Done/` with error annotation; write `ERROR_DATA_*.md` | Human inspects quarantined file |
| **System** (crash, disk full, vault locked) | Watchdog liveness check | Auto-restart (max 3/hour); write `ERROR_SYSTEM_*.md` after threshold | Human notified; manual intervention |
| **Banking timeout** | Any Odoo payment call failure | Write `ERROR_BANK_*.md`; **never** auto-retry | Fresh APPROVAL_*.md required |

### 8.3 Graceful Degradation Fallbacks

| Failure | Fallback Behaviour |
|---|---|
| Email MCP unavailable | Outgoing emails buffered in `scripts/email_outbox_queue.json`; `email_mcp.flush_queue()` called when service restored |
| Claude Code unavailable | All watchers continue running; `Needs_Action/` queue grows; auto-processed when Claude resumes |
| Vault path inaccessible | Watchers write to `VAULT_TEMP_PATH`; orchestrator syncs temp → vault on restoration |
| Social MCP platform API down | `get_post_summary` returns `{ engagement: null }` gracefully; posting retried 3× then `ERROR_SOCIAL_*.md` |
| Odoo unavailable | Finance sync paused; CEO Briefing notes gap; `ERROR_ODOO_UNAVAILABLE.md` written |
| WhatsApp session expired | Single re-auth attempt; on failure, watcher pauses and writes `ERROR_WHATSAPP_SESSION.md` |

### 8.4 Error File Naming Convention

All error files follow: `Needs_Action/ERROR_<CATEGORY>_<PROCESS>_<TIMESTAMP>.md`

Example: `ERROR_AUTH_GMAIL_20260322T143000Z.md`

## 9. Security Design

### 9.1 Secrets Management

All secrets follow the constitution's "Secret Zero" rule:

| Secret | Storage | Usage |
|---|---|---|
| Gmail OAuth2 credentials | `GMAIL_CREDENTIALS_PATH` in `.env` → JSON file on disk (gitignored) | Gmail watcher + Email MCP |
| WhatsApp session | `WHATSAPP_SESSION_PATH` in `.env` → directory on disk (gitignored) | WhatsApp watcher only |
| Odoo admin password | `ODOO_PASSWORD` in `.env` | Odoo MCP only |
| LinkedIn/Facebook/Instagram/Twitter API tokens | `SOCIAL_*_TOKEN` in `.env` | Social MCP only |
| Bank API token | `BANK_API_TOKEN` in `.env` | Finance watcher only |

`.env` is gitignored at the repo root. `.env.example` documents every required key with placeholder values. No secret ever appears in a vault `.md` file, skill file, log file, or committed code.

Credentials are rotated monthly and after any suspected breach (FR-011).

### 9.2 Approval Enforcement

The approval gate is enforced at two levels:

**Level 1 — Skill logic** (FR-017/FR-019): The `process-needs-action` skill evaluates every planned action against the permission boundary table before calling `execute-plan`. Any action categorised as approval-required writes an `APPROVAL_*.md` to `Pending_Approval/` and halts further execution.

**Level 2 — ApprovalWatcher** (structural): `execute-plan` is only invoked by the ApprovalWatcher after it detects a file in `Approved/`. The skill cannot call `execute-plan` for approval-required actions directly — the approval file must exist in `Approved/` first.

No skill may move its own approval file from `Pending_Approval/` to `Approved/` (FR rule + constitution Principle III).

**Payment-specific hardening**: All payment-related MCP tools (`odoo_mcp.post_invoice`) check for an `approval_id` parameter at the tool level. Calls without a valid `approval_id` return an immediate error regardless of caller.

### 9.3 Rate Limiting

Rate limits are enforced inside each MCP server process — they cannot be bypassed by the skill layer:

| MCP | Limit | Enforcement |
|---|---|---|
| email_mcp | 10 `send_email` calls/hour | In-process counter + timestamp window |
| odoo_mcp | 3 `post_invoice`/`create_transaction` calls/hour (payment category) | In-process counter |
| social_mcp | No hard rate limit (platform API limits apply) | Per-platform API rate-limit headers honoured |

On rate limit breach: MCP returns `{ error: "RATE_LIMIT_EXCEEDED", retryable: false }`. Skill writes `ERROR_RATE_LIMIT_*.md` and moves task back to `Needs_Action/` for later retry.

### 9.4 Dry-Run Mode

`DRY_RUN=true` is the default in `.env`. In dry-run mode:
- All MCP `send_*`, `post_*`, `create_*`, `sync_*` tools log the intended action but return `{ dry_run: true, would_have: "<action description>" }` without performing it
- Vault file operations (read, write, move) execute normally
- Audit log entries are written with `"dry_run": true` flag
- Production mode requires explicit `DRY_RUN=false` in `.env`
